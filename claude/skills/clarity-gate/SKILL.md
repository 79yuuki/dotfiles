# Clarity Gate — わかりやすさチェックスキル

## 🎯 設計思想

**「7人に1人が理解できないなら、それは書き手の問題」**

境界知能（IQ 70-84）は日本に約1700万人。知的障害の基準からは外れるため支援対象にならないが、
複雑な文章・専門用語・長い説明は理解が困難。この層にもわかる文章は、全員にとってわかりやすい。

> 参考: 古荘純一『境界知能の人たち』/ gendai.media/articles/-/164706

## いつ使うか

- ✅ 外部公開物（LP、SNS投稿、プレスリリース、ブログ、メール）
- ✅ ユーザー向けドキュメント（README、使い方ガイド、FAQ）
- ✅ 実装内容の説明・提案書・見積書
- ✅ 戦略ドキュメント・decision memo・方針判断メモ
- ✅ team chat報告・ブリーフィング（社内でもわかりやすさは重要）
- ✅ マーケティングコピー全般（muser-marketingと連携）
- ❌ 内部の技術メモ・調査ノート（読者が限定的）
- ❌ コードそのもの（→ code-review）
- ❌ UI/UXのビジュアル面（→ ui-accessibility-design）

## フロー（3ステップ）

### Step 1: 対象の種別判定
アウトプットの種別を判定し、適用する基準セットを選ぶ:
→ [references/output-types.md](references/output-types.md)

| 種別 | 厳しさ | 理由 |
|------|--------|------|
| LP/広告コピー | 🔴 最厳格 | 不特定多数が読む、離脱に直結 |
| SNS投稿 | 🔴 最厳格 | 短時間で理解される必要 |
| ユーザードキュメント | 🟠 厳格 | 操作に影響、誤解は致命的 |
| ブログ/プレスリリース | 🟠 厳格 | 外部公開、ブランドイメージ |
| 提案書/見積書 | 🟡 標準 | 読者は限定的だが理解必須 |
| 戦略ドキュメント / decision memo | 🟡 標準 | 判断の前提・選択肢・見直し条件を明示する必要 |
| team chat報告/社内文書 | 🟢 基本 | 社内だが明瞭さは常に大事 |
| 技術ドキュメント | 🟢 基本 | 専門用語は許容、構造は明瞭に |

### Step 2: チェックリスト実行
→ [references/checklist.md](references/checklist.md)

5カテゴリ × 各3-5項目のチェック:

#### A. 文の構造（Structure）
- [ ] 一文が40文字以内（目安）。60文字超は分割
- [ ] 一段落が3文以内
- [ ] 主語と述語が近い（係り受けが明確）
- [ ] 箇条書き・番号リストを活用している

#### B. 言葉選び（Vocabulary）
- [ ] 専門用語を使う場合は直後に平易な説明がある
- [ ] カタカナ語・英語は必要最小限、日本語の言い換えを併記
- [ ] 二重否定を使っていない
- [ ] 抽象的な表現（「適切に」「適宜」）を具体化している

#### C. 論理の流れ（Flow）
- [ ] 結論が先に来ている（PREP法 or 逆三角形）
- [ ] 「なぜ」「何を」「どうする」が明確
- [ ] 接続詞で論理の流れが明示されている
- [ ] 読者が「次に何をすればいいか」がわかる

#### D. 視覚的わかりやすさ（Visual）
- [ ] 見出し・小見出しで構造化されている
- [ ] 重要な部分が太字・強調されている
- [ ] 適度な空白・改行がある
- [ ] 表や図を使うべき箇所で使っている

#### E. 行動の明確さ（Action）
- [ ] CTA（行動喚起）が具体的（「詳しくはこちら」ではなく「無料で試す」）
- [ ] 手順がある場合は番号付きステップ
- [ ] 期限・条件が明記されている
- [ ] 読者にとってのメリットが先に書かれている

### Step 2.4: 戦略ドキュメント用 Decision Quality add-on
戦略ドキュメント / decision memo / 方針判断メモでは、通常チェックに加えて
→ [references/decision-quality-rubric.md](references/decision-quality-rubric.md)

最低でも以下を確認する:
- 何を決める文書かが冒頭1行でわかる
- 比較した選択肢（現状維持を含む）が並んでいる
- 判断の前提・反証条件・見直し日が明記されている
- 読後に「で、今回は何を採用するの？」で止まらない

### Step 3: 判定と改善提案

#### 判定基準
- **PASS**: 全カテゴリで致命的な問題なし
- **WARN**: 1-2カテゴリで改善推奨（公開可だが改善すべき）
- **FAIL**: 3カテゴリ以上で問題あり、または🔴種別で重大な問題 → 修正必須

#### 出力フォーマット
```
📝 Clarity Gate 結果: [PASS/WARN/FAIL]
対象: [アウトプット種別]
厳しさ: [🔴最厳格 / 🟠厳格 / 🟡標準 / 🟢基本]

✅ Structure: OK
⚠️ Vocabulary: 専門用語3箇所に説明なし
  → 「プロトコル」→「通信の手順（プロトコル）」
  → 「レイテンシ」→「遅延時間（レイテンシ）」
✅ Flow: OK
✅ Visual: OK
⚠️ Action: CTAが抽象的
  → 「お問い合わせください」→「無料相談を予約する（3分で完了）」

改善案: [具体的な書き換え例]
```

## 日本語の読みやすさ指標

→ [references/readability-criteria.md](references/readability-criteria.md)

英語にはFlesch-Kincaid等があるが、日本語には確立された指標が少ない。
本スキルでは以下の実用的基準を採用:

- **平均文長**: 40文字以下（目標）、60文字以下（許容）
- **漢字含有率**: 20-30%（新聞レベル）。40%超は読みにくい
- **専門用語密度**: 100文字あたり1語以下（🔴種別の場合）
- **受動態比率**: 20%以下
- **一文一意**: 1文で伝えることは1つ

## 他スキルとの連携

### muser-marketing との連携
マーケティングコピー生成時、muser-marketing → clarity-gate の順で適用。
- LP/広告コピーは自動で 🔴最厳格 基準を適用
- CTA の明確さチェックはこのスキルが担当
- コピーライティングの「刺さる表現」は muser-marketing が担当（わかりやすさと別軸）

### fact-check-gate との連携
- fact-check-gate → 事実の正確さ
- clarity-gate → 表現のわかりやすさ
- 両方通すのが理想（特に外部公開物）

### x-cmo との連携
- X投稿は文字数制限があるため、構造チェックは簡略化
- 専門用語チェックと行動明確さチェックは通常通り適用

## 🚫 アンチAIパス（Step 2.5 — 外部公開物で必須）

> ソース: Wikipedia Signs of AI Writing + blader/humanizer（2026-03-24追加）

Step 2のチェックリスト通過後、🔴🟠種別の外部公開物に対して実行:

**AI頻出パターン検出（日本語）:**
- [ ] 「さらに」「加えて」が2回以上連続していない
- [ ] 「〜として位置づけられています」「〜の役割を果たしています」→ 「〜です」に
- [ ] 「〜を浮き彫りにしており」「〜に大きな示唆を与えています」→ データの後に意義づけを付け足さない
- [ ] 「今後の展開が注目されます」「いかがでしたでしょうか」→ 禁止
- [ ] 三点セット強制（A、B、そしてC）パターンがない
- [ ] emダッシュ（—）・全角ダッシュの不自然使用がない
- [ ] **太字: キーワード** のインラインヘッダー箇条書きがない（人間はこの形式をあまり使わない）
- [ ] 冒頭が「ここでは〜について解説します」で始まっていない
- [ ] 結論が「今後の展開が注目されます」で終わっていない
- [ ] 「お世辞的トーン」がない（「素晴らしい」「画期的な」の乱用）

**対処:** 検出したパターンは具体的な書き換え例を提示。

**参考:** blader/humanizer (GitHub) / humanizer_academic / Wikipedia:Signs_of_AI_writing

## クイックチェック（簡易版）

全項目が不要な場合の5秒チェック:
1. **中学生が読んでわかるか？** （境界知能の基準に近い）
2. **声に出して読んで引っかからないか？**
3. **「つまり何？」に一文で答えられるか？**
4. **次に何をすればいいかわかるか？**
5. **一番大事な情報が最初にあるか？**
6. **AIが書いたっぽくないか？**（アンチAIパス）
