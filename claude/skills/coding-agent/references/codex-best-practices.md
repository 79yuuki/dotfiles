# OpenAI Codex ベストプラクティス（公式ドキュメント準拠）
> Source: developers.openai.com/codex/learn/best-practices + codex/prompting + codex/workflows (2026-03)
> derrickcchoi の投稿がきっかけで公式ドキュメント化された内容

## 1. プロンプトの4要素

毎回のタスク指示に以下を含める：

| 要素 | 説明 | 例 |
|------|------|-----|
| **Goal** | 何を変更/構築するか | 「認証モジュールをリファクタ」 |
| **Context** | 関連ファイル、ドキュメント、エラー | `@auth.ts @schema.ts` |
| **Constraints** | 規約、安全要件、制約 | 「public APIは変更しない」 |
| **Done when** | 完了条件 | 「テスト通過、リンター0エラー」 |

**Muserでの適用:** orchestrator から投げる task や coding-agent の prompt に必ずこの4要素を入れる。

## 2. 計画先行（Plan-first）

複雑なタスクは実装前に計画を立てさせる：

- **Plan mode:** `/plan` または Shift+Tab で計画モードに
- **インタビュー方式:** 「私のアイデアに質問して、前提を崩してから具体化してくれ」
- **PLANS.md テンプレート:** 長期タスクには実行計画テンプレートを使う

**Muserでの適用:** dual-agent-dev / agent-teams-dev のブレインストーミングステップで活用。

## 3. 自律性と持続性（Autonomy & Persistence）

Codex公式プロンプトのコア思想：

- **自律的シニアエンジニアとして動け** — 方向が決まったら、コンテキスト収集→計画→実装→テスト→改善を自分で回す
- **タスクが完全に終わるまで止まるな** — 分析や部分修正で止まらない。実装→検証→説明まで一気通貫
- **行動バイアス** — 合理的な仮定で実装する。質問で終わるな（本当にブロックされてる時だけ聞け）
- **無限ループ回避** — 同じファイルを読み直し/再編集して進展がないなら止めて要約+質問

## 4. コード実装の原則

| 原則 | 詳細 |
|------|------|
| **正確性 > 速度** | 危険なショートカット、投機的変更、汚いハックは避ける |
| **コードベース準拠** | 既存パターン・ヘルパー・命名・フォーマット・ローカライゼーションに従う |
| **網羅性** | 関連する全サーフェスを調査・接続し、アプリ全体で動作が一貫するように |
| **エラーハンドリング** | 広い try/catch 禁止、成功風フォールバック禁止、エラーは明示的に伝播 |
| **型安全** | `as any` / `as unknown as ...` 禁止。型ガードと既存ヘルパーを使う |
| **DRY/検索ファースト** | 新しいヘルパーを追加する前に既存コードを検索。共有ヘルパーを抽出 |

## 5. 探索と並列読み込み

**公式推奨パターン:**

```
1. まず考える — 必要な全ファイル/リソースを決定
2. バッチで読む — 複数ファイルは一度に並列読み込み
3. 結果を分析
4. 新たに必要なら繰り返す
```

**禁止:** ファイルを一つずつ逐次読み込み（論理的に不可避な場合を除く）

**Muserでの適用:** agent-teams-dev の探索フェーズで特に重要。各チームメイトに「並列読み込みで探索しろ」と明示指示する。

## 6. 検証ループ（Verify before done）

コード変更後の必須チェック：

1. テスト作成/更新
2. テストスイート実行
3. リント・フォーマット・型チェック
4. 最終動作が要件と一致するか確認
5. diff でバグ・リグレッション・リスクパターンを確認

**Muserでの適用:** Codex Review Gate の前段として、コーディングエージェント自身にもこのループを回させる。

## 7. AGENTS.md 設計（公式ガイドライン）

### 含めるもの
- リポジトリ構造と重要ディレクトリ
- プロジェクト実行方法
- ビルド・テスト・リントのコマンド
- エンジニアリング規約とPR期待値
- 制約と禁止事項
- 完了条件と検証方法

### やってはいけないこと
- 長く曖昧なルールの羅列（短く正確なものが有効）
- 繰り返しミスが起きてから追加するのが正解（先回りしすぎない）
- 大きくなりすぎたら分割（タスク別mdファイルに参照）

### 改善ループ
**同じミスが2回起きたら → Codexにレトロスペクティブさせて → AGENTS.mdを更新**

## 8. Git安全規則

- `git reset --hard` / `git checkout --` は明示的に要求されない限り使わない
- 自分がやっていない既存の変更を勝手にリバートしない
- コミットは明示的に要求されない限りamendしない
- 作業中に予期しない変更を見つけたら → 即停止してユーザーに確認

## 9. フロントエンド品質（AI slop回避）

AI生成UIが陥りがちなパターンを回避：

- **タイポグラフィ:** Inter/Roboto/Arial/system はデフォルト逃げ。意図的なフォント選択を
- **カラー:** 紫×白のデフォルトバイアスを避ける。CSS変数で方向性を定義
- **背景:** フラット単色禁止。グラデーション、シェイプ、パターンで雰囲気を
- **全体:** ボイラープレートレイアウト禁止。テーマ・書体・ビジュアル言語にバリエーションを

## 10. Reasoning Level 使い分け

| レベル | 用途 |
|--------|------|
| Low | 高速・スコープ明確なタスク |
| Medium | 一般的なコーディング（推奨デフォルト） |
| High | 複雑な変更・デバッグ |
| Extra High | 長時間の自律的推論タスク |

## 11. ワークフロー別レシピ

### コード理解
```
このモジュールの責務と、他モジュールとの連携を説明して。
含めてほしいもの:
- 各モジュールの責務の要約
- データバリデーションの場所
- 変更時の注意点1-2個
```

### バグ修正（再現レシピ付き）
```
Bug: [症状]
Repro:
1) [手順1]
2) [手順2]
Constraints:
- APIの形は変えない
- 最小限の修正 + リグレッションテスト
まず再現してから、パッチを提案してチェックを実行して。
```

### UI プロトタイプ（画像 + 制約）
```
このスクリーンショットを元に新しいダッシュボードを作成。
Constraints:
- React, Vite, Tailwind, TypeScript
- スペーシング・タイポグラフィ・レイアウトを可能な限り再現
Deliverables:
- 新しいルート/ページ
- 必要なコンポーネント
- ローカル実行手順のREADME
```

### イテレーティブUI改善
```
ランディングページの改善案を2-3個提案して。
→ [選択後] オプション2で。ヘッダーだけ変更:
- タイポグラフィをもっとエディトリアルに
- 余白を増やす
- モバイルでも良く見えること
```
