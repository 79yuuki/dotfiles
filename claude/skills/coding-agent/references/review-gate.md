# Codex Review Gate（必須）

コーディングエージェントの作業が完了したら、必ずCodexによる **多観点レビュー** を実施。
OKが出るまで修正→再レビューをループする。

> 💡 Anthropic公式 pr-review-toolkit（6専門エージェント方式）を参考に、
> Codex単体で多観点レビューを実現する設計。

## フロー

```
1. コーディングエージェント（Claude Code / Codex / Pi）が作業完了
2. git diff で変更内容を取得
3. Codex に6観点レビューを依頼（別セッション）
4. Codex が問題を指摘 → コーディングエージェントに修正指示を送る
5. 修正完了 → 再度 Codex レビュー
6. Codex が「LGTM」→ 完了
```

## 6観点レビュー（Codex に渡すプロンプト）

```bash
DIFF=$(git diff HEAD)

codex exec "以下のdiffを6つの観点でレビューしてください。
各観点ごとに問題があれば具体的なファイル名・行番号・修正案を出してください。
問題がなければその観点は「OK」と記載。
全観点OKなら最後に「LGTM」と出力。

## レビュー観点

### 1. コード品質（code-quality）
- CLAUDE.md/AGENTS.md のコーディング規約への準拠
- 命名規則、コードスタイル、DRY原則
- 既存パターンとの一貫性

### 2. バグ・ロジックエラー（bug-detection）
- ロジックの誤り、境界値エラー、off-by-one
- null/undefined の未処理
- 競合状態、デッドロック

### 3. エラー処理（error-handling）
- broad catch / silent failure がないか
- エラーが適切に伝播されているか
- フォールバック処理の妥当性

### 4. テストカバレッジ（test-coverage）
- 新規コードに対応するテストがあるか
- エッジケース・エラーケースのテスト
- テストの品質（意味のあるアサーション）

### 5. セキュリティ（security）
- インジェクション脆弱性（SQL, XSS, コマンド）
- 認証・認可の漏れ
- 機密情報のハードコード
- 依存関係の既知脆弱性

### 6. パフォーマンス（performance）
- N+1クエリ、不要なループ
- メモリリーク
- 不要な再レンダリング（フロントエンド）

---
\$DIFF
"
```

## 信頼度スコア（推奨）

Codexに各指摘に信頼度（0-100）を付けさせる:
- **90-100**: 確実に修正すべき → 即修正
- **70-89**: 高確度で問題 → 修正推奨
- **50-69**: 可能性あり → 要確認
- **0-49**: 誤検知の可能性 → スキップ可

プロンプトに追加: `各指摘に信頼度スコア(0-100)を付けてください。80未満は報告のみ、80以上は修正必須としてください。`

## バックグラウンド自動ループ

1. **コーディング完了を検知**（process:log で完了確認）
2. **diff取得** → Codex 6観点レビュー起動
3. **レビュー結果をパース**
   - `LGTM`（全6観点OK）→ 完了報告
   - 信頼度80以上の指摘あり → まとめてコーディングエージェントに再投入
   - 信頼度80未満のみ → 報告はするが通過可
4. **最大3ループ**で打ち切り（無限ループ防止）

## ルール
- レビューは作業ディレクトリとは別の場所でCodexを起動（コンテキスト汚染防止）
- レビュー対象は `git diff` ベース（変更差分のみ）
- 「LGTM」「OK」「問題なし」「No issues found」で通過
- 3ループ超えたら強制停止 → ユーザーに報告
- レビュー結果は毎回ログに残す
- 2ループ以上で通過した場合 → lessons.md にパターンを記録（lessons-gate連携）
