# 記事量産パイプライン

## 二本柱: 新規記事 + リライト

月110本の実績は「全部ゼロから書いた」のではなく:
- **40本: 新規記事**（ニュース解説、ツール使い方ガイド、ノウハウ）
- **70本: リライト/情報更新**（料金変更、機能追加、サービス終了の反映）

この比率（新規36% : リライト64%）がポイント。既存資産の更新が量の大部分を占める。

## 新規記事パイプライン

### ソース → 記事の変換パターン

| ソース | 記事タイプ | 例 |
|---|---|---|
| AIニュース | 速報解説 | 「GPT-5が発表。何が変わるのか」 |
| SNSで反応良かった投稿 | 深掘り記事 | 「投稿で書いた○○を詳しく解説」 |
| ツール検証 | 使い方ガイド | 「△△の使い方を解説」 |
| 自社の開発ログ | 裏側記事 | 「こうやって仕組みを作った」 |
| 顧客の課題 | ハウツー記事 | 「□□で困ってる人向けガイド」 |

### 記事構成テンプレート（速報解説）
```
1. リード: 何が起きたか（1-2文）
2. 背景: なぜ重要か
3. 詳細: 具体的に何が変わるか
4. 影響: ユーザー/業界への影響
5. 所感: 自分の意見（差別化ポイント）
```

### 記事構成テンプレート（使い方ガイド）
```
1. ○○とは（概要）
2. 料金プラン
3. 始め方（ステップバイステップ）
4. 主要機能の使い方
5. メリット/デメリット
6. 他ツールとの比較
7. まとめ
```

## リライト/情報更新パイプライン

### 陳腐化検出の仕組み

1. **料金変更検出**: ツールの料金ページを定期クロール → 差分検出
2. **機能追加検出**: リリースノート/ブログのRSS監視
3. **サービス終了検出**: HTTPステータス監視 + ニュースウォッチ
4. **日付ベース検出**: 「2024年最新」→ 2025年になったら要更新

### リライト優先度判定

| シグナル | 優先度 | 理由 |
|---|---|---|
| 料金変更（値上げ/値下げ） | 高 | 読者の購買判断に直結 |
| サービス終了 | 高 | 存在しないサービスの記事は信頼毀損 |
| 大型機能追加 | 中 | 記事の価値が相対的に低下 |
| 日付の陳腐化 | 低 | 内容が正しければ影響小 |
| 軽微なUI変更 | 低 | スクショ更新程度 |

### 自動化可能な部分

- 料金テーブルの更新 → APIまたはクロールで自動取得
- 「最新」年号の更新 → 正規表現で一括置換
- サービス終了の検出 → HTTP監視 + アラート
- リライト候補のリストアップ → 週次バッチ

### 手動が必要な部分

- 記事の文脈に合わせた書き換え
- 新機能の評価・使用感
- 競合ツールとの比較更新
- 「自分らしさ」の追加

## 双方向展開

### SNS → 記事（ボトムアップ）
1. SNS投稿のエンゲージメントを確認
2. 反応が良いもの → 記事化候補に追加
3. 記事化: 140文字を2000-5000文字に膨らませる
4. 記事公開 → SNSで再告知

### 記事 → SNS（トップダウン）
1. 記事のキーメッセージを抽出
2. プラットフォーム別に短縮・整形
3. 記事リンク付きで投稿
4. スレッド/カルーセルで要約展開

## KPI

| 指標 | 目安 |
|---|---|
| 月間新規記事数 | 30-50本 |
| 月間リライト数 | 50-80本 |
| 記事あたり作成時間（新規） | 30-60分 |
| 記事あたり作成時間（リライト） | 10-20分 |
| 記事→SNS展開率 | 100%（全記事をSNSに） |
| SNS→記事化率 | 10-20%（反応良いもののみ） |
