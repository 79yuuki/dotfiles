# UI Audit Process — デザイン監査プロセス

既存UIの問題を体系的に発見・優先順位付け・改善するプロセス。

---

## フェーズ概要

```
Phase 1: スコープ定義（30分）
Phase 2: ヒューリスティック評価（2-4時間）
Phase 3: アクセシビリティ自動チェック（1時間）
Phase 4: ユーザー観察（オプション、2-4時間）
Phase 5: 問題の優先順位付け（1時間）
Phase 6: 改善提案書作成（2-3時間）
```

---

## Phase 1: スコープ定義

**やること**:
1. 監査対象のページ・コンポーネントをリストアップ
2. ビジネス目標とユーザー目標を確認
3. 既知の問題点・制約を把握（エンジニア・ビジネス側から）
4. 監査の成果物を定義（レポート形式・提案の深さ）

**x402での例**:
- LP: CVRが低い → CTAまわりの認知負荷を重点監査
- Directory: 離脱率が高い → 検索→詳細フローを監査
- Hub: サポート問い合わせが多い → エラー状態・フォームを監査

---

## Phase 2: ヒューリスティック評価

Nielsenの10原則に基づいて評価。各項目をGood/Warning/Badで記録。

### チェック項目

**1. システム状態の可視性**
- ローディング状態はわかるか
- 成功/失敗がすぐにわかるか
- 現在位置がわかるか（パンくずリスト・ステップ表示）

**2. 実世界との一致**
- ユーザーが理解できる言葉を使っているか
- メタファーが直感的か
- 日付・数値フォーマットがローカライズされているか

**3. ユーザーコントロールと自由度**
- 間違いを簡単に取り消せるか
- 危険な操作に確認ステップがあるか
- キャンセル・戻るが常に使えるか

**4. 一貫性と標準**
- 同じ操作に同じUI要素を使っているか
- プラットフォームの慣習（iOS/Android/Web）に従っているか
- アイコンの使い方が統一されているか

**5. エラー防止**
- ユーザーがエラーを犯しにくい設計か
- 危険な操作へのアクセスは制限されているか
- フォームのバリデーションがリアルタイムか

**6. 記憶への負担を減らす**
- 前のステップの情報を再入力させていないか
- コンテキストが画面内に保持されているか
- ユーザーが覚えなくてもいい情報を表示しているか

**7. 柔軟性と効率**
- パワーユーザー向けのショートカットがあるか
- よく使う操作にすぐアクセスできるか

**8. 美学と最小デザイン**
- 関連性のない情報が画面を占有していないか
- 視覚的ヒエラルキーが明確か
- 「上品なノイズ」（不要な装飾）がないか

**9. エラーリカバリー**
- エラーメッセージが具体的で行動可能か
- 解決策を提示しているか

**10. ヘルプ・ドキュメント**
- ヘルプへのアクセスが一貫した場所にあるか
- コンテキストに合ったヘルプが提供されているか

## Phase 3: アクセシビリティ自動チェック

**ツール**:
1. **axe DevTools**（Chrome拡張）: 自動でa11y問題を検出
2. **Lighthouse**（Chrome DevTools > Lighthouse > Accessibility）: スコア確認
3. **WAVE**（wave.webaim.org）: 視覚的なa11yオーバーレイ
4. **Colour Contrast Analyser**: 任意の箇所のコントラスト比を測定

**手動確認**:
1. Tabキーだけでページを操作してみる（フォーカス順序確認）
2. スクリーンリーダーで読み上げてみる（VoiceOver/NVDA）
3. ブラウザのズームを200%にして確認
4. OSのハイコントラストモードで確認

**記録方法**:
```
問題ID | 要素 | 問題内容 | WCAG基準 | 重大度
A001  | Submit button | コントラスト比 2.8:1（必要: 7:1） | 1.4.6 | Critical
A002  | Error message | aria-live未設定 | 4.1.3 | High
```

## Phase 4: ユーザー観察（オプション）

**5名で十分**（Nielsen Norman Groupの研究より）。

### セッション設計

```
1. タスクシナリオを3-5つ作成（実際のユースケースから）
2. 「声に出して考えながらやってください」と指示
3. 介入しない（詰まっても助けない → それが発見）
4. 困惑・迷い・バックトラックを記録
5. 完了後に「どこが難しかったですか？」
```

**被験者選定（多様性優先）**:
- 年齢: 20代・40代・60代を含む
- 技術リテラシー: 高/中/低を含む
- デバイス: PC/スマホを含む
- 対象サービスの既存ユーザー + 新規ユーザー

**記録すべきこと**:
- ✅ タスク完了率・時間
- ✅ エラー発生箇所・頻度
- ✅ 「えっと...」「なんだこれ」発言
- ✅ マウス/指の迷い（予期しない場所を触る）
- ✅ 「修正不要」な部分（スムーズに完了した部分も記録）

---

## Phase 5: 問題の優先順位付け

### 重大度分類

| レベル | 定義 | 対応時期 |
|---|---|---|
| Critical | 目標達成不可能 / アクセス不可 | 即座に修正 |
| High | 重大な困難・失敗が頻発 | 次スプリント |
| Medium | 使えるが不快・非効率 | バックログ優先 |
| Low | 軽微な改善点 | 時間があれば |
| Enhancement | 現状問題なし、改善余地あり | 将来検討 |

### 優先度マトリクス（Impact × Effort）

```
高Impact × 低Effort → Quick Win（最優先）
高Impact × 高Effort → Major Project（計画して実施）
低Impact × 低Effort → Fill In（空き時間で）
低Impact × 高Effort → Thankless Task（後回しor見送り）
```

## Phase 6: 改善提案書

**構成テンプレート**:

```markdown
## UIデザイン監査レポート: [プロダクト名]
実施日: YYYY-MM-DD
対象: [ページ/コンポーネント一覧]

### エグゼクティブサマリー
- 発見した問題数: Critical X / High X / Medium X / Low X
- 最優先対応: [3つ以内]
- 推定対応工数: [日数]

### Critical課題
#### [C001] [問題タイトル]
- **現状**: スクリーンショット/コード
- **問題**: 具体的に何が問題か
- **影響**: 誰にどんな影響があるか
- **改善案**: 具体的な修正内容（Before/After）
- **工数**: [時間]

### アクセシビリティスコア
- axe DevTools: X問題
- Lighthouse Accessibility: XX/100
- コントラスト違反: X箇所

### 「修正不要」な部分
ユーザーテストでスムーズだった箇所（変更しないこと）
```

## x402プロダクト別 監査フォーカス

### LP（ランディングページ）
**優先確認**:
1. ファーストビューのCTAが1つに絞られているか
2. Hero → Value Prop → CTA の流れが自然か
3. モバイルでCTAが親指で届く位置にあるか
4. コントラスト比（特にHero上のテキスト）

### Directory
**優先確認**:
1. 検索 → 結果 → 詳細のフロー一貫性
2. 空検索・0件状態のUI
3. フィルター・ソートのキーボード操作
4. カード一覧のスクリーンリーダー読み上げ順序

### Hub
**優先確認**:
1. フォームのエラーメッセージ（具体性）
2. 操作完了後のフィードバック（成功/失敗）
3. ダッシュボードの視覚的ヒエラルキー
4. 設定変更の確認フロー
