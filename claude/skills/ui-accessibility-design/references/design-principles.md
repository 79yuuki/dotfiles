# Design Principles — UI Accessibility Design

Cloudflare Turnstileリデザイン事例をベースにした実践的デザイン原則。

---

## 1. 統一された情報アーキテクチャ（One IA）

**原則**: 複数のコンポーネント/画面で同じ構造パターン・視覚ヒエラルキー・配置を使う。

**Turnstileでの実践**:
- Turnstile Widget（インライン小型）とChallenge Pages（フルページ）で同じIA構造を採用
- ロゴ配置・ステータス表示・アクションボタン・ヘルプリンクの位置を統一
- ユーザーはどちらの画面でも同じ認知パターンで処理できる

**適用方法**:
1. UIコンポーネントの「族」を定義する（例: モーダル族、カード族、フォーム族）
2. 各族で共通のIA構造テンプレートを作る
3. 新コンポーネント追加時は必ずテンプレートに従う
4. 逸脱する場合は意図的に・理由を明記

**注意**: IAの統一は「見た目の統一」ではなく「構造の統一」。スタイルが変わっても構造は保つ。

---

## 2. WCAG AAA準拠

**原則**: 全ユーザーが使えるUI。年齢・能力・文化・技術リテラシーを問わない。

**AAとAAAの違い**:
| 基準 | AA（最低要件） | AAA（理想） |
|---|---|---|
| テキストコントラスト | 4.5:1 | 7:1 |
| 大テキストコントラスト | 3:1 | 4.5:1 |
| フォーカス表示 | 表示されること | 高コントラスト |
| エラー提案 | 可能なら提示 | 必ず具体的提案 |

**Turnstileでの実践**:
- 8カ国・多様な年齢層・技術リテラシーで検証
- 高齢者・スクリーンリーダーユーザー・低コントラスト環境も考慮
- 「誰でも使える」＝「全員に最高の体験」ではなく「誰も排除しない」

**優先対応順**:
1. コントラスト比（視覚障害・低照度環境）
2. キーボード操作（運動障害・パワーユーザー）
3. スクリーンリーダー対応（視覚障害）
4. 認知負荷軽減（認知障害・非技術者）
5. 言語・文化対応（グローバルユーザー）

---

## 3. Don't Make Me Think（認知負荷最小化）

**原則**: ユーザーが「考えなくても」使えるUI。迷わせない。

**Steve Krugの核心**:
- 明白な選択肢を1つだけ提示する
- 説明が必要なUIはデザインが失敗している
- 慣習（Convention）は破る理由がある時だけ破る

**実践チェック**:
- ユーザーが「次に何をすればいいか」が一瞬でわかるか？
- 選択肢が3つ以上あれば絞れないか検討する
- ページ上の最重要アクションは1つだけか（Primary CTA）

**アンチパターン**:
- 「詳しくはこちら」「もっと見る」だけで何が見られるか不明
- 同じページに同等の重要度を持つCTAが複数存在
- エラーメッセージが「エラーが発生しました」のみ

---

## 4. 一貫性の力

**原則**: 同じ操作・同じ状態には同じパターンを使う。

**統一すべき要素**:
- **エラー状態**: 表示位置・アイコン・色・コピーのトーン
- **空状態**: イラスト有無・メッセージ・CTAの有無
- **ローディング状態**: スピナー vs スケルトン、配置
- **成功状態**: トースト vs インライン、表示時間
- **ヘルプリンク**: 常に右下 or 常にヘッダー（どこかに固定）

**Turnstileでの実践**:
- エラー状態のアイコン・テキスト・「再試行」ボタンの配置を全チャレンジタイプで統一
- ヘルプリンクは常に同じ場所
- ブランドロゴは常に同じ位置

---

## 5. ユーザーリサーチの原則

**原則**: データと観察で意思決定する。仮説だけで進めない。

**Turnstileの手法**:
- 8カ国・8名の被験者（多様性優先）
- ABテスト（新旧デザイン比較）
- タスクベース観察（「このページで何をしますか？」）

**重要な発見**: 「修正不要な部分」の発見が重要
- ユーザーが困っていない部分は変えない
- 変更コスト（学習コスト・バグリスク）を払う価値があるか検討
- リデザインは「全部変える」ではなく「問題のある部分だけ変える」

**小規模でできるリサーチ**:
1. 5名の観察セッション（廊下テスト）
2. ヒートマップ分析
3. フォーム入力でのドロップオフ箇所の分析
4. サポート問い合わせの分類（何で詰まっているか）

---

## 6. コピーライティング原則

**原則**: UIのテキストは「明確」「人間的」「行動可能」の3つを満たす。

**避けるべきパターン**:
- `Error 403: Access Forbidden` → `このページにアクセスする権限がありません`
- `Processing...` → `確認中です（数秒かかります）`
- `Are you sure?` → `削除すると元に戻せません。削除しますか？`

**フィードバック選択肢の明確化**（Turnstileの事例）:
- 旧: `このサイトの問題を報告`
- 新: `問題を具体的に選択` → `チャレンジが表示されない` / `ループする` / `その他`

**ルール**:
1. 専門用語（HTTP status code、エラーコードなど）をそのまま出さない
2. エラーメッセージは「何が問題か」＋「どうすればいいか」をセットで
3. 確認ダイアログは「はい/いいえ」ではなく「[動詞]/キャンセル」（例: 削除する/キャンセル）

---

## 7. 制約はデザインの友

**原則**: 選択肢を制限することで、本当に重要なディテールに集中できる。

**Turnstileの制約**:
- ウィジェットサイズが固定 → 配置・優先度を強制的に整理できた
- 改ざんできないCF要素 → そこに集中せず変えられる部分に注力

**適用方法**:
1. 制約（技術・時間・予算）を最初にリストアップ
2. 制約内でできる「最善」を定義する
3. 制約を嘆かず、制約がフォーカスを生むと捉える

**x402への適用**:
- LPは「1スクロールでCVするか否か」がKPIとして制約になる
- Directoryは「検索→詳細の2ステップ以内」が設計制約
- Hubは「ユーザーの主要タスクを3クリック以内」が制約
