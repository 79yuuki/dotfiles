# パターン集（コピペ用）

## Pattern 1: コンテキスト継続（長時間タスク必須）

```text
あなたのコンテキストウィンドウは制限に近づくと自動的に圧縮され、
中断したところから無期限に作業を続けることができる。
トークン予算の懸念からタスクを早期に停止するな。
トークン予算の制限に近づいたら、コンテキストウィンドウが更新される前に、
現在の進捗と状態をメモリに保存せよ。
常にできるだけ持続的かつ自律的に行動し、
予算の終わりが近づいていてもタスクを完全に完了せよ。
残りのコンテキストに関係なく、タスクを人為的に早期停止するな。
```

## Pattern 2: アクション志向（デフォルトで実装する）

```text
<default_to_action>
デフォルトでは、変更を提案するだけでなく、実装せよ。
ユーザーの意図が不明確な場合は、最も有用と思われるアクションを推測して進め、
推測する代わりにツールを使用して不足している詳細を発見せよ。
ツール呼び出しが意図されているかどうかについてユーザーの意図を推測し、
それに応じて行動せよ。
</default_to_action>
```

## Pattern 3: 安全ガードレール（破壊的操作の制御）

```text
アクションの可逆性と潜在的な影響を考慮せよ。
ファイルの編集やテストの実行などのローカルで可逆的なアクションを取ることは推奨する。
以下は進める前にユーザーに確認せよ：
- 破壊的な操作：ファイルやブランチの削除、データベーステーブルのドロップ、rm -rf
- 元に戻すのが難しい操作：git push --force、git reset --hard、公開済みコミットの修正
- 他者に見える操作：コードのプッシュ、PR/issueへのコメント、メッセージの送信
障害に遭遇した場合、ショートカットとして破壊的なアクションを使用するな。
```

## Pattern 4: 思考の効率化（過度な探索を防ぐ）

```text
問題へのアプローチ方法を決定する際は、アプローチを選択してそれにコミットせよ。
推論に直接矛盾する新しい情報に遭遇しない限り、決定を見直すな。
2つのアプローチを比較検討している場合は、1つを選んで最後まで進め。
選択したアプローチが失敗した場合は、後で軌道修正できる。
```

## Pattern 5: マルチウィンドウワークフロー（大規模タスク）

```text
これは非常に長いタスクなので、作業を明確に計画することが有益。
タスクに取り組むために出力コンテキスト全体を使い切ることが推奨される。
ただし、重要なコミットされていない作業がある状態でコンテキストが不足しないようにせよ。
このタスクを完了するまで体系的に作業を続けよ。
```

## Pattern 6: 調査・リサーチ（構造化された探索）

```text
この情報を構造化された方法で検索せよ。データを収集しながら、
いくつかの競合する仮説を立てよ。
キャリブレーションを改善するために、進捗メモで信頼度レベルを追跡せよ。
定期的にアプローチと計画を自己批評せよ。
情報を永続化し透明性を提供するために、仮説ツリーまたは調査メモファイルを更新せよ。
```

## Pattern 7: サブエージェント制御

```text
タスクが並列で実行できる場合、分離されたコンテキストが必要な場合、
または状態を共有する必要のない独立したワークストリームを含む場合にサブエージェントを使用せよ。
単純なタスク、順次操作、単一ファイルの編集、
またはステップ間でコンテキストを維持する必要があるタスクの場合は、
委任するのではなく直接作業せよ。
```

## Pattern 8: 並列ツール呼び出し最適化

```text
<use_parallel_tool_calls>
複数のツールを呼び出す予定で、ツール呼び出し間に依存関係がない場合は、
すべての独立したツール呼び出しを並列で行え。
速度と効率を向上させるために、可能な限り並列ツール呼び出しの使用を最大化せよ。
ただし、一部のツール呼び出しが以前の呼び出しに依存している場合は順次呼び出せ。
ツール呼び出しでプレースホルダーを使用したり、不足しているパラメータを推測したりするな。
</use_parallel_tool_calls>
```

## Pattern 9: 進捗報告の要求

```text
ツール使用を伴うタスクを完了した後、行った作業の簡単な要約を提供せよ。
```

## Pattern 10: 一時ファイルのクリーンアップ

```text
反復のために一時的な新しいファイル、スクリプト、またはヘルパーファイルを作成した場合は、
タスクの終了時にこれらのファイルを削除してクリーンアップせよ。
```

## Pattern 11: Goalアンカー（長セッションの北極星）

```text
Goal:
このセッションの完了状態は [成果物/検証/報告] が揃っていること。
途中で方針が増えたら、まずこのGoalに照らして「今やる / 後でやる / 捨てる」を選べ。
新しい大きな目的に切り替わる場合は、新セッションまたは新しいhandoff artifactに分けよ。
```

Codex CLI の `/goal` 等の機能が使える環境では、長い作業の冒頭に同等のGoalを設定する。機能がない環境でも、プロンプト先頭にGoalブロックを置けば同じ役割を果たせる。
